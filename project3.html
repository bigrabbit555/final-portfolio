Iterative Design

Context

This group project sought to emphasize the iterative aspect of the design process, asking students not only to design an interface from the ground up but also to gather feedback about it and make improvements to it.

My group decided to design a web interface for 1Flow, a platform that allows developers to receive in-platform feedback on mobile and web apps. After designing our hi-fi prototype, we received user feedback from fellow students, after which we iterated on our design to make it more user-friendly and improve user experience. We submitted our prototype to usertesting.com to receive additional feedback on our design, which we reflected on to consider what additional changes could be made for the next iteration.

We decided to implement a website because user testing feedback is not necessarily a service that would need to be accessed on-the-go. A website, which would more effectively display statistics, is a more intuitive platform for the company. We included what we considered to be the most essential functionalities in our mockup. Our prototypes include the ability to edit questionnaires that would be shown to the test users as well as a “User View” page so developer s can see what their users see. We also included a “Stats” page that displays relevant statistics such as click-through rates in a graphical way along with users' answers to the questionnaire.

Initial Sketches

To get ideas flowing, each group member made three sketches outlining what they thought were some of the key pages and functionality for their vision of the site.


Low-Fidelity Prototypes with Balsamiq

After some discussion as a group, we combined all the best parts about each person's sketches to create low-fidelity wireframes of our site using the program Balsamiq. These would serve as the basis for site architecture and functionality, as well as the layout of the high-fidelity prototypes to come.

User Testing

Having received feedback from peers and iterated once already, it was time to get some real-world opinions from people who weren't already familiar with the project. As mentioned above, we did this through usertesting.com, submitting our interactive Figma prototype with instructions to perform the following tasks: sign into the website, interact with the questionnaire settings (edit a question, add a question, delete a question), and view the statistics page and "User View" pages.

The most common error was with trying to find the "Questionnaire Edit" page—since the button was just labeled "Edit", ⅔ users felt that was confusing and weren't sure what exactly that button would have them editing. This is likely in part due to the inconsistency in labeling between the instructions and the prototype, but clarifying the express purpose of the "Edit" button can only help with clarity. Another user also recommended some kind of "instructions" page or popup that would remind a user how to use the site and what options are available to them. Based on this feedback, we could potentially add an instructions page and change the edit button to read "Edit Questionnaire".

Conclusion

As useful as tools like personas and storyboards are for getting into the heads of users and designing effective interfaces from the getgo, it's nigh impossible to get things perfect on the first try. This project was enlightening in that it showed me what to do when things aren't perfect: how to gather various types of feedback from various sources, and how to convert that feedback into action with regard to the interface. Going forward I think that I'll be much more comfortable embracing imperfection with these practices under my belt, and that I'll be able to achieve more successful interface design with the iterative process.
